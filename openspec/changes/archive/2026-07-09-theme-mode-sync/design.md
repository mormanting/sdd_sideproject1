## Context

「筆境」目前已經有固定的編輯骨架、即時預覽與 Mermaid 解析流程。這一階段要補的是主題層的能力：同一份內容在淺色與深色模式下都要維持可讀性，且切換後不能讓程式碼高亮或圖表主題停留在舊狀態。

限制條件是既有頁面已以單一 `index.html` 與原生 JavaScript 實作，且先前已引入 Tailwind、Highlight.js 與 Mermaid。這意味著主題切換必須在不引入新框架的情況下完成，並儘量沿用現有渲染管線。

## Goals / Non-Goals

**Goals:**
- 提供可持久化的淺色 / 深色模式切換。
- 讓 HTML root class、Tailwind 樣式、Highlight.js 主題與 Mermaid 主題同步。
- 切換主題時，保留目前編輯內容並立即重繪預覽。
- 在頁面初始化時優先套用使用者最近一次選擇。

**Non-Goals:**
- 不新增第三種主題或自訂配色編輯器。
- 不改變 Markdown 編輯器的核心輸入行為。
- 不處理跨裝置同步，只使用本機 `localStorage`。
- 不在此階段重構整個渲染管線或拆分檔案結構。

## Decisions

1. **以 `<html>.dark` 作為單一主題開關**
   - 理由：Tailwind 的深色模式支援已能直接跟 `dark` class 對接，這讓全站配色切換最直接。
   - 替代方案：在各元件上單獨切換 class。
   - 為何不選：容易造成狀態分散，也增加漏改風險。

2. **把 `localStorage.theme` 視為主題狀態來源**
   - 理由：需求明確要求持久化，且這個資料量很小，不需要更重的狀態層。
   - 替代方案：使用 cookie 或 URL query。
   - 為何不選：cookie 與此需求無關；URL query 不適合記憶偏好設定。

3. **Highlight.js 樣式表採用固定 `id` 並直接切換 `href`**
   - 理由：CSS 主題是單一資源，直接替換 link 最簡單，且可避免載入多個主題互相覆蓋。
   - 替代方案：同時載入兩個主題並靠 selector 隱藏。
   - 為何不選：會增加下載與瀏覽器樣式判定負擔。

4. **Mermaid 在主題切換時重新初始化後重繪**
   - 理由：Mermaid 主題是初始化時決定的，切換時最穩定的做法就是重新設定 theme 並重新渲染現有圖表。
   - 替代方案：只改 CSS，保留原圖。
   - 為何不選：Mermaid SVG 的線條與文字顏色常不是單靠外層 CSS 就能完整修正。

5. **切換動畫只做顏色過渡，不改變版面**
   - 理由：150ms 的 `transition-colors` 足以降低閃爍感，不會讓版面響應變慢。
   - 替代方案：更長的動畫或加入其他轉場效果。
   - 為何不選：會干擾寫作流暢度，且沒有必要。

## Risks / Trade-offs

- [主題切換瞬間重繪成本高] → 重新初始化 Mermaid 會有額外成本；緩解方式是只在切換主題時執行，不在一般輸入時觸發。
- [CDN 主題資源載入失敗] → 高亮主題依賴外部 CSS；緩解方式是在失敗時保留可用的基礎樣式與主題按鈕狀態。
- [localStorage 被停用] → 若瀏覽器阻擋儲存，主題偏好無法持久化；緩解方式是回退到預設淺色模式。
- [Mermaid 舊圖殘留樣式] → 若未清空舊渲染結果，可能出現混色；緩解方式是在重繪前清理預覽區中的 Mermaid 容器。

## Migration Plan

1. 在 HTML head 中為 Highlight.js 樣式表與 Mermaid 初始化預留可切換位置。
2. 加入主題狀態讀寫與按鈕控制，先完成 UI 與持久化。
3. 讓高亮主題與 Mermaid 主題跟著狀態切換並重繪。
4. 最後補上初始化順序與回退處理，確保首次載入與切換後都一致。

Rollback 方式是保留淺色模式作為安全預設，並在主題切換管線出錯時停用深色 class、恢復 github CSS 與 Mermaid default 主題。
