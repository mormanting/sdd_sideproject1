## Context

「筆境」已經有編輯、主題與圖表聯動能力，但目前 UI 文字還是以單一語系硬編碼。這會讓後續語系擴充必須碰 HTML、元件結構與多個事件處理點，不利於維護，也容易在新增翻譯時漏改。

本階段要建立的不是一次性的翻譯替換，而是一個可持續擴充的 i18n 架構。首波只支援繁體中文與 English，但未來要能透過補字典的方式延伸到其他語系，而不需要重寫頁面結構。

## Goals / Non-Goals

**Goals:**
- 讓使用者在繁體中文與 English 之間即時切換，且不需重新整理頁面。
- 將全站顯示文案抽離到 JavaScript 字典，透過 `data-i18n` 對應注入。
- 使用 `localStorage.lang` 記憶語系偏好，並在初始化時優先套用。
- 讓導覽列、工具列、狀態列、主題控制提示、滑桿旁說明與 Mermaid 錯誤訊息都能被翻譯。

**Non-Goals:**
- 不導入外部 i18n 套件或翻譯服務。
- 不做 server-side locale negotiation。
- 不處理 RTL 版面、字型 fallback 或日期時間格式化。
- 不在這一版做複數規則或區域化數字格式處理。

## Decisions

1. **採用字典驅動的單一翻譯來源**
   - 理由：目前所有文字都集中在單一頁面，字典是最低成本、也最容易擴充的做法。
   - 替代方案：把文案散落在各個元件內自行管理。
   - 為何不選：會讓未來新增語系時必須掃多個位置，容易漏翻或不一致。

2. **以 `data-i18n` 做 DOM 對應，不直接在 HTML 寫死顯示文案**
   - 理由：HTML 只保留語意與結構，文字由腳本注入，能保證切換時整頁同步更新。
   - 替代方案：在 HTML 裡保留預設文字，再由 JS 局部覆寫。
   - 為何不選：會殘留 hard-code，和需求中的資料驅動要求衝突。

3. **語系狀態以 `localStorage.lang` 為唯一持久化來源**
   - 理由：偏好設定只需要本機記憶，不需要額外資料層。
   - 替代方案：cookie 或 URL 參數。
   - 為何不選：cookie 與此場景不匹配，URL 也不適合保存 UI 偏好。

4. **翻譯更新採用一次性遍歷與同步注入**
   - 理由：頁面規模小，直接遍歷 `data-i18n` 元素即可做到即時且可理解的更新。
   - 替代方案：引入狀態管理庫或 virtual DOM。
   - 為何不選：過度設計，對單頁 MVP 沒有實質收益。

5. **把 Mermaid 錯誤訊息也納入 i18n 管線**
   - 理由：這是面向使用者的提示文字，必須跟其他 UI 一起翻譯，避免出現混語。
   - 替代方案：錯誤訊息維持固定英文或中文。
   - 為何不選：會破壞整體語系一致性。

## Risks / Trade-offs

- [HTML 文字逐步遺留 hard-code] → 若未完整標記 `data-i18n`，可能出現混語；緩解方式是將翻譯掃描納入驗證清單。
- [字典 key 漏配] → 新增語系時若缺少 key，畫面可能出現空白；緩解方式是提供 fallback 與明確的 default locale。
- [初始化順序衝突] → i18n 需要先於畫面互動綁定完成，否則會短暫看到錯誤語系；緩解方式是把語系套用放在 bootstrap 早期。
- [未來語系複雜度上升] → 這版不處理複數、語境或性別等複雜語言特性；緩解方式是保留字典擴充接口與 key 命名一致性。

## Migration Plan

1. 先建立語系字典與 `data-i18n` 對應規則，讓文字可以被集中管理。
2. 再把導覽列、工具列與底部狀態列換成字典驅動的渲染。
3. 接著補上語系選單與 `localStorage.lang` 持久化。
4. 最後把主題提示與 Mermaid 錯誤提示一起接進 i18n 管線，完成全站覆蓋。

Rollback 方式很直接：保留繁體中文字典作為預設來源，若翻譯映射出問題，將語系選單固定回 `zh` 即可恢復可用狀態。

## Open Questions

- 語系選單是否需要顯示語言原名，例如 `繁體中文` 與 `English`，還是採用更短標籤，目前尚未需要再拆細。
- 未來新增語系時是否要把字典拆到獨立模組，還是維持在單一檔案，目前可以先不決定。
