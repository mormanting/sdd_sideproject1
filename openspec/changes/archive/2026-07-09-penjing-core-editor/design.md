## Context

「筆境」這一版的目標不是做完整產品，而是先把高專注度寫作環境的核心骨架站穩。現階段同時包含版面結構、文字輸入互動、即時預覽、多種語法渲染與 Mermaid 圖表效能控制，因此需要先把技術邊界與渲染策略定清楚，再進入實作。

限制條件很明確：單一 `index.html`、原生 JavaScript、透過公共 CDN 載入外部庫、預設淺色模式與繁體中文。這使得方案更偏向前端即用型 MVP，而不是可擴充的模組化應用架構。

## Goals / Non-Goals

**Goals:**
- 建立可直接使用的雙欄 Markdown 編輯與即時預覽骨架。
- 支援基礎 Markdown、程式碼高亮與 Mermaid 向量圖表渲染。
- 在 Mermaid 更新上加入 300ms 防抖，避免輸入過程中頻繁重繪。
- 讓字體縮放、文字統計與工具列插入動作在單一頁面內完成。
- 保持 Light Mode 與繁體中文作為預設體驗。

**Non-Goals:**
- 不處理檔案持久化、版本控制、同步或導出流程。
- 不引入打包器、框架或伺服器端渲染。
- 不做多人協作、即時共享或權限模型。
- 不在此階段追求完整的語法擴充或主題系統。

## Decisions

1. **採用單一 `index.html` 搭配原生 JavaScript**
   - 理由：需求明確要求單檔 MVP，且這能把啟動成本壓到最低，方便後續直接打開使用。
   - 替代方案：使用 Vite / React / Vue 之類的應用框架。
   - 為何不選：會引入不必要的建置流程與專案結構，和本階段的目標不符。

2. **使用 `marked` + `marked-highlight` + `highlight.js` + `mermaid` 組成渲染管線**
   - 理由：這組工具剛好覆蓋需求中的基礎 Markdown、語法高亮與 Mermaid。
   - 替代方案：自行寫 Markdown parser，或改用單一大型編譯器。
   - 為何不選：自行實作風險高且成本大；單一大型編譯器會讓能力分層不清楚，調整空間也較差。

3. **採用 `textarea` 作為編輯器，而不是 `contenteditable`**
   - 理由：游標插入、快捷語法包裹、統計與同步預覽都更可預期。
   - 替代方案：`contenteditable` 或自製富文字編輯器。
   - 為何不選：`contenteditable` 的游標與換行行為在純 Markdown 編輯場景下較難控制，也更容易引入錯誤狀態。

4. **將 Mermaid 重繪與一般 Markdown 更新分開處理**
   - 理由：一般 Markdown 可以即時更新，但 Mermaid 重繪成本高，應只在使用者停止輸入後觸發。
   - 替代方案：所有內容同步即時重繪。
   - 為何不選：會造成不必要的重繪壓力，與需求中明確的效能防護牆衝突。

5. **以 CSS 變數承接字體縮放**
   - 理由：同一個滑桿要同步影響編輯區與預覽區，CSS 變數最直接、也最容易維持一致。
   - 替代方案：逐節點手動更新字級或重建 DOM。
   - 為何不選：維護成本高，且容易產生不同區塊字級不一致。

## Risks / Trade-offs

- [CDN 依賴失效] → 主要外部庫全走 CDN，若來源不可用會直接影響功能；緩解方式是維持降級提示與最小可用版面。
- [Mermaid 重繪延遲感] → 300ms 防抖會讓圖表更新不是「每鍵即時」；這是刻意取捨，以換取輸入流暢度。
- [單頁規模膨脹] → 所有邏輯集中在單一檔案，未來擴充時容易變大；緩解方式是先把渲染、統計、插入與狀態更新拆成獨立函式。
- [Mobile 切換狀態複雜] → 行動裝置需在編輯/預覽間切換，可能造成狀態不同步；緩解方式是維持單一來源資料與明確的檢視模式狀態。

## Migration Plan

1. 先交付靜態骨架與基本排版，確認版面與裝置切換行為。
2. 接上 Markdown 解析與程式碼高亮，驗證一般文字內容可穩定預覽。
3. 補上 Mermaid 轉換與 300ms 防抖，針對效能與重繪行為做檢查。
4. 最後再收斂字體縮放、文字統計與工具列插入等互動細節。

Rollback 方式很直接：保留單檔入口與 CDN 引用，若某段渲染邏輯有問題，可先停用對應功能並退回基礎 Markdown 預覽。
